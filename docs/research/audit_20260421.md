**KilaBz consistently misses:**

- Missing error logging on state mutations. Several files suppress stderr silently and KilaBz didn't flag it.
- No review cross-referenced whether a flagged finding had already been fixed by the validate.sh wiring. Review 9 flagged ACCESS_LEVEL failing open — that's already fixed in current code (line 178 aborts on parse failure). A stale finding still consumes engineer attention.

**Antman:** No Antman reviews in this batch. Cannot assess.

### Top 3 Improvements

1. **KilaBz system prompt — add "verify current code" instruction:** Before rating a finding, KilaBz should check whether the flagged line already has a fix in the same file (e.g., a corrective comment, a subsequent guard). This prevents stale auto-review findings from landing as P1s on already-patched code. Add: *"If a finding appears to already be mitigated by surrounding code, note the mitigation and downgrade to P3 or PASS."*

2. **KilaBz system prompt — require one "behavioral observation":** Beyond line-level findings, require one structural observation per review: e.g., "this hook runs N times per edit due to X." This captures the duplicate-trigger class of bugs that no individual line finding catches.

3. **Process — auto-review hook dedup before dispatch:** The hook is filing 2-3 reviews per edit of the same file (Reviews 1+12, 2+13, 5+14 are exact duplicates). KilaBz can't fix this — the dedup state file is the problem. The hook's own P2 on missing `flock` on the dedup file is causing it to miss its own dedup gate.
